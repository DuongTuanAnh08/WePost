UAT: Admin Dashboard & Management

1. Lời nói đầu (Feature Definition)

  • Feature Name: Admin Dashboard & Management
  • Objective: Tạo trang dành riêng cho Admin để quản lý tổng quan hệ thống (số lượng truy cập, tài khoản), kiểm duyệt nội dung (quản lý/xóa bài đăng) và xử lý vi phạm (khóa/ban người dùng).
  • User Story: Là một Admin, tôi muốn xem thống kê số liệu hoạt động và danh sách người dùng/bài đăng để có thể xóa bài viết vi phạm hoặc cấm (ban) các tài khoản gửi spam, bảo vệ cộng đồng.
  • Security Impact: Yêu cầu xác thực phân quyền mạnh (Role-based Authorization) tại Server (Custom Filter/Servlet). User thường không thể truy cập dù có Link. Các thao tác Ban/Delete yêu cầu HTTP POST bảo mật.
  • Database Impact:
    • Cần thêm trường dữ liệu IsBanned BIT DEFAULT 0 vào bảng Users hiện tại.
    • Sử dụng cơ chế Application-level context hoặc tạo bảng DB con để theo dõi "Lượt truy cập trang web" (Session Count).
  • Performance Impact: Truy vấn đếm (COUNT) cần dùng Index và tối ưu. Lấy dữ liệu Bảng cần lấy các khóa chính cần thiết thay vì SELECT * để tiết kiệm RAM phía Java.
  • Rollback Impact: Dễ rủi ro thấp. Có thể thu hồi tính năng bằng cách ngắt Servlet Mapping.
  • Risk Level: Cao (Quyền lực mạnh có thể làm thay đổi hệ thống và dữ liệu người khác).

2. Phạm vi (Scope)

  1. Admin Dashboard UI: Giao diện có màu sắc tách biệt (Dark/Red/Blue) để không nhầm với trang Feed. Có các thẻ thống kê: Tổng Users, Tổng Bài Đăng, Hệ đếm Lượt truy cập web.
  2. Quản trị User (Moderation): Bảng hiển thị người dùng, trạng thái, có nút Ban / Unban xử lý AJAX Real-time.
  3. Quản trị Post: Bảng hiển thị list bài đăng, có nút Xóa thẳng khỏi Database.

3. Functional Test Cases (Các ca kiểm thử)

Test ID Mô Tả Kết quả mong muốn
FTC01 Tài khoản Role = 'ADMIN' truy cập link /admin Truy cập thành công, giao diện thống kê tải đầy đủ.
FTC02 Tài khoản Role = 'USER' hoặc chưa Login truy cập /admin Chặn truy cập, ép chuyển hướng về /home hoặc /login (Forbidden).
FTC03 Admin bấm "Khóa (Ban)" Tài khoản User X Tài khoản X bị đổi trạng thái IsBanned = 1, User X khi đăng nhập sẽ dính thông báo "Bị khóa".
FTC04 Admin bấm "Xóa" một bài đăng Y Bài đăng Y bị xóa khỏi Database, biến mất trên trang chủ mọi người dùng.

4. Các khía cạnh khác (Edge, Boundary, Negative, Security)

  • Edge Cases: Admin ấn nhầm nút Khóa chính bản thân mình (Hệ thống phải che đi nút khóa đối với row danh tính của mỉnh, chống Autocide). Đếm Visit sẽ được tính theo Base trên Session ID để tránh Spam F5 liên lục tăng Page view ảo.
  • Boundary Conditions: Số lượng User/Post lấy về hàng nghìn record. UI áp dụng thanh lăn overflow-y: auto với max-height cố định.
  • Negative Testing: Fake POST request lên Server /admin/ban?userId=2 qua Postman với User bình thường không có cờ Admin. Response trả về là lỗi Server cự tuyệt.
  • Security Testing: Sử dụng hệ thống Filter Servlet đón chặn (Interceptor) đứng che chở nguyên đường dẫn `/*` thay vì chỉ giấu nút của front-end HTML.
  • Rollback Plan: Bỏ trường Cột IsBanned mới thêm khỏi Data nếu xảy ra sự cố logic không thể cứu vẫn. Dừng Servlet Web.xml không cho route Admin chạy.
