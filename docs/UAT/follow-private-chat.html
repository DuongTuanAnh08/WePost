Tài liệu Kiểm Thử Người Dùng (UAT)

Tính năng: Theo dõi Người dùng & Nhắn tin riêng tư 1-1

1. Mục tiêu (Objective)

  • Xây dựng chức năng cho phép bấm "Theo dõi" (Follow) nhau trên trang Hồ sơ.
  • Hai người cùng theo dõi nhau sẽ trở thành Bạn bè và có thể chat riêng tư.
  • Cải tiến hệ thống WebSockets từ Chat chung thành Chat 1-1 (Private Message) bảo mật cao.

2. Cốt truyện người dùng (User Story)

  • Người dùng vào Hồ sơ của người khác để bấm Theo dõi / Hủy Theo Dõi.
  • Mở trang Tin nhắn và lấy danh sách Bạn bè chuẩn xác bên cột trái.
  • Khi nhấp vào tên một người bạn, tin nhắn được tải và có thể chat Real-time với riêng người đó. Người ngoài không thấy được nội dung.

3. Phạm vi công việc (Scope)

  • Java Backend: Tạo FriendDAO, cập nhật ProfileServlet (Xử lý nút bấm Theo dõi), ChatServlet (Load Friend list), và ChatEndpoint (Xử lý Endpoint socket chat 1-1 mapping Session).
  • Frontend JSP: Chỉnh sửa Javascript WebSockets ở chat.jsp từ Broacast sang Gửi đúng Tên người nhận. Bắt sự kiện Click nút Follow ở giao diện profile.jsp.

4. Ca kiểm thử chức năng (Functional Test Cases)

  • TC01: Người A bấm Theo dõi người B => Trạng thái lưu DB là A Following B.
  • TC02: Người A bấm gửi Hủy Theo dõi => Thao tác Database Hủy bỏ Row Follow/Update Status thành công.
  • TC03: A và B cùng thiết lập theo dõi nhau => Sang Chat được mở khóa giao diện "Bạn Bè", thấy tên đối diện bên thanh Sidebar.
  • TC04: 2 Browser (1 Chrome thường, 1 Chrome ẩn danh) nhắn với nhau => Tin nhảy Real-time chính xác 1-1, log Console Server Java hiển thị truyền chéo.
  • TC05: Các Message gửi/nhận phải được lưu vào mảng MessageHistory riêng cho cặp A-B, không bị lộ ra ngoài với C.

5. Biên kiểm thử & Trường hợp Ngoại lệ (Edge/Boundary/Negative Testing)

  • Người dùng dùng Inspect / Postman cố tình ép lệnh Follow chính tài khoản mình => Database/Backend cấm không cho vào (Bắt Exception SQL hoặc chặn IF).
  • Giới hạn màn hình chat tải danh sách tối đa 50-100 bạn bè để chống sập DOM trình duyệt trên điện thoại yếu.
  • Người dùng A dùng Hacker scripts gửi gói WebSocket nặc danh (Sửa Tên người gửi là B, gửi cho C) => Endpoint phải thẩm định chéo bằng HTTP Session có khớp Tên đăng nhập gốc hay không trước khi chạy lệnh.

6. Tác động - Hiệu suất và Bảo mật (Impact)

  • Bảo mật: Cần xử lý kĩ thuật Encoding JSON chống lỗ hổng XSS vào các Tin nhắn cá nhân. Đếm cẩn thận session.getId().
  • Cơ sở dữ liệu: Tạo bảng mapping dữ liệu tăng tải vào bảng Friends dựa vào File database.sql sẵn có của dự án.
  • Hiệu suất: ServerEndpoint chuyển từ 1 vòng lặp Set đơn thuần sang ConcurrentHashMap<String, Session> để bắt Map Tên Người Dùng -> Màn hình Đang Mở.
  • Mức độ rủi ro: Cao (High Risk - Level Core Override) do tháo dỡ hoàn toàn luồng chat Global cũ và móc chặt vào SQL Server Friends Table.
  • Rollback: Khôi phục File code WebSockets thuần gốc tại commit dự án gần nhất.
